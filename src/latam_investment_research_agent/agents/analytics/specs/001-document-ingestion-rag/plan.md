## Summary

Build two compiled LangGraph graphs — an **Ingestion Graph** and a **RAG Query Graph** — that
together form the analytics agent framework for extracting numerical data from LATAM financial
PDFs and web pages, storing it in ClickHouse with smart LLM-based table routing, and enabling
natural-language querying of the stored data with structured CSV exports.

The ingestion graph fetches a document, uses `gpt-4o-mini` to extract numerical datasets, routes
each dataset to an existing or newly created ClickHouse table via LLM judgment, and returns a
structured ingestion summary. The RAG query graph introspects the ClickHouse schema, selects
relevant tables via LLM, assembles and executes SQL queries, and exports results to CSV via
Polars with rationale and SQL returned to the caller.

**Language/Version**: Python 3.12+

**Primary Dependencies**:
- `langgraph` — LangGraph agent graph orchestration
- `langchain-core` — `BaseChatModel` LLM provider abstraction
- `langchain-openai` — OpenAI provider (default: `gpt-4o-mini`)
- `clickhouse-connect` — Official async ClickHouse Python driver
- `pdfplumber` — Financial PDF table extraction (MIT, pure Python)
- `httpx` — Async HTTP client for PDF URL fetch and web page scraping
- `beautifulsoup4` — HTML table parsing
- `polars` — High-performance CSV export
- `pydantic` (v2) — Data validation and LLM structured output schemas
- `pytest` + `pytest-asyncio` — Test framework
- `mypy` (strict) — Type checking
- `ruff` — Linting and formatting

**Storage**: ClickHouse (OLAP, columnar) — dynamically created `MergeTree()` tables

**Testing**: `pytest` with `pytest-asyncio`; 90% coverage requirement

**Target Platform**: Python library (LangGraph graph objects) invoked by parent orchestrator

**Performance Goals**:
- 100-page PDF ingested end-to-end in ≤5 minutes (SC-001)
- RAG query returns CSV + rationale in ≤30 seconds (SC-004)

**Constraints**:
- All I/O MUST be async
- No ORMs — raw ClickHouse SQL via `clickhouse-connect`
- LLM provider swappable via env vars (no code changes)
- Read-only ClickHouse access in the RAG query graph

**Scale/Scope**: Single document per ingestion run; up to 10 ClickHouse tables per RAG query

## Architecture Decisions

### LangGraph Graph Compilation

Both graphs are compiled once at module load time as module-level singletons. This avoids
repeated compilation overhead on each invocation, which is significant for LangGraph's
StateGraph compilation step.

```python
# graph/ingestion_graph.py — compiled at import time
ingestion_graph = build_ingestion_graph()
```

→ **See**: `graph/ingestion_graph.py` and `graph/rag_query_graph.py` in the source structure.
→ **State schemas**: [data-model.md § LangGraph State Schemas](data-model.md#langgraph-state-schemas)

### Ingestion Graph: Dataset Loop Pattern

LangGraph handles the per-dataset routing loop via a conditional edge from
`write_to_clickhouse_node` back to `route_dataset_node` while
`state["current_dataset_index"] < len(state["extracted_datasets"])`. When the index is
exhausted, the conditional edge routes to `build_ingestion_summary_node`.

→ **State field driving the loop**: `IngestionState.current_dataset_index` —
  [data-model.md § IngestionState](data-model.md#ingestionstate-internal--ingestion-graph)
→ **Full node sequence with loop diagram**: [contracts/ingestion_graph.md § Node Sequence](contracts/ingestion_graph.md#node-sequence)
→ **Error routing from fetch failure**: [contracts/ingestion_graph.md § Error Behaviour](contracts/ingestion_graph.md#error-behaviour)

### LLM Provider Factory

`create_llm_provider(config: AnalyticsConfig) -> BaseChatModel` reads `config.llm_provider`
and `config.llm_model_name` and returns the appropriate LangChain chat model. Nodes receive
the `BaseChatModel` via dependency injection in the node constructor — they never instantiate
the LLM directly.

→ **Rationale for BaseChatModel interface**: [research.md § LLM Provider Abstraction](research.md#llm-provider-abstraction)
→ **Config fields**: `AnalyticsConfig` in `config.py`; env var names in `.env.sample`

### LLM-Based Numerical Extraction: Prompt & Schema Design

The `numerical_extractor.py` service sends document text (in page-batches of 5) to
`gpt-4o-mini` with a Pydantic v2 strict-mode response schema. The `ExtractedDataset`
domain model is the direct Pydantic target. Key design constraints:

- Use `Decimal` fields (not `float`) for monetary values in the Pydantic extraction schema
  to avoid floating-point precision loss. Map to ClickHouse `Decimal(18, 4)` type.
- System prompt MUST include locale-aware number parsing instructions (period vs. comma
  as decimal separator — common in Portuguese-language LATAM financial documents).
- Retry up to 3 times on schema validation failure, appending the validation error to the
  prompt on retry.
- Page batching: feed pdfplumber pages in batches of 5 to stay within model context limits.

→ **Rationale**: [research.md § LLM-Based Numerical Data Extraction](research.md#llm-based-numerical-data-extraction)
→ **ExtractedDataset schema**: [data-model.md § ExtractedDataset](data-model.md#extracteddataset)

### LLM-Based Table Routing: Prompt Design

The `table_router.py` service sends (a) dataset name, context labels, and column structure
plus (b) existing table names and column schemas to `gpt-4o-mini`. It returns a
`RoutingDecision` Pydantic model with `target_table_name` set to an existing table name
or the sentinel `"__create_new__"`, plus `routing_action` and `rationale`.

The `__create_new__` sentinel MUST be the canonical string checked by the repository.
If `routing_action == "create"`, `proposed_schema` contains the `ColumnDefinition` list
for the new table including all data columns (audit columns are always added by the
repository, not proposed by the LLM).

→ **Rationale for LLM routing over rules/embeddings**: [research.md § LLM-Based Table Routing](research.md#llm-based-table-routing)
→ **RoutingDecision and ColumnDefinition schemas**: [data-model.md § RoutingDecision](data-model.md#routingdecision)
→ **Repository handling of create vs append**: see ClickHouse Repository section below

### ClickHouse Repository: Schema Evolution

When `route_dataset_node` returns `routing_action = "create"`, the repository:
1. Prepends mandatory audit columns (`source_reference`, `ingestion_timestamp`,
   `content_hash`) to the LLM-proposed column list — the LLM MUST NOT propose these.
2. Executes `CREATE TABLE IF NOT EXISTS` with `MergeTree()` engine,
   `ORDER BY (source_reference, content_hash, ingestion_timestamp)`.

When `routing_action = "append"` and the dataset introduces columns absent from the
existing schema, the repository issues `ALTER TABLE {table} ADD COLUMN {col} {type}
DEFAULT ''` for each new column before inserting.

→ **Mandatory audit column definitions**: [data-model.md § ClickHouse Table Conventions](data-model.md#clickhouse-table-conventions)
→ **ClickHouse schema strategy rationale**: [research.md § ClickHouse Schema Strategy](research.md#clickhouse-schema-strategy)

### Deduplication Strategy

Per-row content hash: `hashlib.sha256(json.dumps(row, sort_keys=True, default=str).encode()).hexdigest()`
(full 64-character hex digest — do NOT truncate). Stored in `content_hash String` column.
Insert uses `INSERT INTO {table} ... WHERE (source_reference, content_hash)
NOT IN (SELECT source_reference, content_hash FROM {table})` pattern.

→ **content_hash field in state**: [data-model.md § ExtractedDataset](data-model.md#extracteddataset)
  (note: the content_hash is computed per-row at write time, not per-dataset)
→ **Idempotency guarantee**: [contracts/ingestion_graph.md § Idempotency](contracts/ingestion_graph.md#idempotency)

### RAG Query: SELECT-Only Guard

Before executing any LLM-generated query, `execute_queries_node` validates that each query
string begins with `SELECT` (case-insensitive, stripped). Non-SELECT queries are logged as
errors and skipped — the graph never executes them.

→ **Read-only guarantee specification**: [contracts/rag_query_graph.md § Read-Only Guarantee](contracts/rag_query_graph.md#read-only-guarantee)
→ **RAGQueryState.assembled_sql_queries**: [data-model.md § RAGQueryState](data-model.md#ragquerystate-internal--rag-graph)

### Polars CSV Export

`export_results_node` uses `polars.from_dicts(query_result_records).write_csv(export_file_path)`.
File path: `{export_directory}/{YYYYMMDD_HHMMSS}_{question_slug}.csv`.
If `was_truncated` is `True`, this is noted in the `rationale` field of the output.

→ **File naming convention**: [contracts/rag_query_graph.md § CSV File Naming](contracts/rag_query_graph.md#csv-file-naming)
→ **RAGQueryOutput.was_truncated field**: [data-model.md § RAGQueryOutput](data-model.md#ragqueryoutput-public--returned-to-caller)
→ **Polars rationale**: [research.md § CSV Export](research.md#csv-export)
